Conceitos Fundamentais

PCB (Process Control Block)

Estrutura que guarda o "estado" de cada processo: valores do acumulador (ACC), ponteiro de instrucao (PC), estado atual (Pronto, Executando, Bloqueado) e parametros de escalonamento.

EDF (Earliest Deadline First)

Politica de escalonamento dinamico e preemptivo. A cada instante, a tarefa com o deadline absoluto mais proximo ganha o processador. Se uma tarefa nova chega com deadline menor, ela interrompe a atual.

Periodo (Pi) e Deadline

A cada Pi unidades de tempo, a tarefa "renasce" com um novo deadline absoluto = tempo_atual + Pi. Neste trabalho, deadline = periodo. A tarefa precisa completar Ci instrucoes antes do deadline.

Troca de Contexto

Quando o processador troca de tarefa, salva ACC e PC da tarefa atual no PCB e restaura os valores salvos da proxima tarefa. Assim cada tarefa retoma de onde parou.

Bloqueio (I/O)

SYSCALL 1 (imprimir) e SYSCALL 2 (ler) simulam operacoes de I/O. O processo fica bloqueado por 1 a 3 unidades de tempo (aleatorio), liberando o processador para outra tarefa.

Modos de Enderecamento

Imediato (#): o operando e o valor literal. Direto (sem #): o operando e o indice na memoria de dados cujo conteudo sera usado. Ex: ADD #5 soma 5; ADD x soma o valor de x.

Parser de Arquivos

Ler .code/.data, tokenizar instrucoes, resolver labels
Base ▼
Conceito Didatico — Por que um Parser?

Um processador real nao entende texto. Ele entende instrucoes binarias. O simulador precisa fazer o mesmo: transformar LOAD variable em algo como um objeto Instruction("LOAD", "variable", DIRETO). Isso e o que compiladores e montadores fazem na vida real. O parser faz duas passadas: a primeira mapeia labels para indices, a segunda resolve as referencias.

Tarefas de implementacao

Criar classe Parser que le arquivo linha a linha
Identificar secoes .code/.endcode e .data/.enddata
Extrair mnemonico, operando e modo de enderecamento (# = imediato)
Primeira passada: mapear labels (ex: ponto1:) para indice da instrucao
Segunda passada: substituir referencias a labels pelos indices
Secao .data: mapear variaveis para posicoes na memoria com valor inicial

Base ▼
Conceito Didatico — Ciclo Fetch-Execute

Todo processador real opera num loop: busca a instrucao apontada pelo PC, decodifica qual operacao fazer, executa a operacao, e incrementa o PC. O simulador faz exatamente isso, so que em Java e uma instrucao por "unidade de tempo". Pense no PC como o "dedo" que aponta para a linha atual do programa.

Implementar saltos: BRANY, BRPOS, BRZERO, BRNEG

Modelo de Processos (PCB)

Nucleo ▼
Conceito Didatico — PCB (Process Control Block)

No SO real, quando o processador troca de processo, salva o estado atual (registradores, PC) numa estrutura chamada PCB. Quando o processo volta a executar, o SO restaura esses valores. Sem isso, o processo "perderia a memoria" de onde parou. Pense no PCB como um "marcador de livro" para cada processo.

4

Escalonador EDF

Fila de prontos por deadline, preempcao, reativacao periodica
Critico ▼
Conceito Didatico — Como o EDF funciona

Imagine 3 tarefas: A (deadline=10), B (deadline=7), C (deadline=5). O EDF escolhe C primeiro (deadline mais proximo). Quando C termina, escolhe B. Se no meio da execucao de B aparece D com deadline=4, D preempta B (toma o processador). E "dinamico" porque a prioridade muda: a cada periodo, a tarefa ganha um novo deadline = tempo_atual + Pi.

Tarefas de implementacao

Fila de prontos: PriorityQueue<PCB> ordenada por deadline absoluto
A cada tick: verificar novas chegadas (arrival time == tempo atual)
Reativacao periodica: se tempo_atual e multiplo de Pi, reativar tarefa
Na reativacao: novo deadline = tempo_atual + Pi, resetar Ci e PC
Preempcao: se tarefa na fila tem deadline menor que a tarefa em execucao, trocar
Deteccao de perda de deadline: tempo_atual >= deadline e remainingCi > 0
Teste: simular 2+ tarefas com periodos diferentes e validar a timeline

// Logica central do escalonador por tick:
void tick(int currentTime) {
  // 1. Chegadas e reativacoes
  checkArrivals(currentTime);
  checkPeriodicReactivations(currentTime);

  // 2. Escolher tarefa (menor deadline)
  PCB next = readyQueue.peek();

  // 3. Preempcao?
  if (running != null && next.deadline < running.deadline) {

  }

  // 4. Executar 1 instrucao
  cpu.executeOneInstruction(running);
  running.remainingCi--;
}

SYSCALL e Bloqueio

Chamadas de sistema, I/O simulado, timer de bloqueio
Nucleo ▼
Conceito Didatico — Chamadas de Sistema e I/O

Quando um processo precisa fazer I/O (imprimir ou ler do teclado), ele nao pode continuar executando — precisa esperar o dispositivo. No SO real, o processo fica "bloqueado" ate o I/O completar. No simulador, isso e representado por um timer aleatorio de 1 a 3 unidades de tempo. Enquanto bloqueado, outra tarefa pode usar o processador.

Tarefas de implementacao

SYSCALL 0: finalizar tarefa no periodo (halt)
SYSCALL 1: imprimir acc no terminal + bloquear [1,3] ut
SYSCALL 2: ler inteiro do teclado para acc + bloquear [1,3] ut
Mover processo para BLOQUEADO com timer aleatorio
A cada tick: decrementar timers. Se timer == 0, mover para PRONTO
Reinserir na fila EDF ao desbloquear

Arquitetura de Classes (Visao Geral)

Parser

Le os arquivos de programa. Produz objetos Program com listas de Instruction e o mapa de memoria de dados.

CPU

Executa instrucoes uma a uma. Recebe o PCB da tarefa em execucao, usa seu ACC/PC, e retorna o resultado (normal, syscall, halt).

PCB

Representa um processo: estado, contexto salvo, parametros EDF (Pi, deadline, Ci), timer de bloqueio.

Scheduler

Gerencia a fila de prontos (PriorityQueue por deadline), decide preempcao, trata reativacoes e detecta deadline miss.

SyscallHandler

Interpreta SYSCALL 0/1/2, executa I/O, gerencia transicao para estado bloqueado e timers.

SimulationEngine (main)

Loop principal do simulador. Coordena todos os modulos a cada unidade de tempo. Coleta e imprime resultados.
